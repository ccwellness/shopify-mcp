DESIGN REQUIREMENTS
Shopify Multi-Store Data Platform
Python / Flask connector with REST + MCP interfaces for lubelife.com, shopjo.com, shopshibari.com
Version 0.1 (Draft) · April 29, 2026

1. Executive Summary
This document specifies the design of an internal Python/Flask application that consolidates order, catalog, inventory, subscription, and analytics data from three Shopify stores — lubelife.com, shopjo.com, and shopshibari.com — into a single Postgres-backed system. The application exposes data through three consumption surfaces: an internal web dashboard, a REST/GraphQL API for other internal tools, and an MCP server that lets Claude (and other LLM agents) answer natural-language questions across the three stores.
Authentication is handled via a Custom App provisioned in each store, granting a per-store Admin API access token. Data is kept fresh through a hybrid sync model: Shopify webhooks push order, inventory, and product events in near real time, while an on-demand fetch path falls back to live GraphQL queries when the local cache is stale or a query covers data that is not synced (e.g., ad-hoc historical reports). A nightly reconciliation job uses the Bulk Operations API to repair any events missed during webhook delivery failures.
The design is portable: it runs on a single developer laptop with Postgres in Docker and a Cloudflare/ngrok tunnel for webhooks, but is structured (12-factor configuration, stateless web layer, environment-driven secrets) so it can be lifted into any cloud target without code changes when the team is ready.
2. Goals and Non-Goals
2.1 In Scope
Pull and persist full order detail across all three stores: line items (units, SKU, item number), revenue (gross, net, tax, discount, shipping), shipping address, financial status, fulfillment status, processed-at, and order name.
Pull and persist catalog data: products, variants, options, prices, media URLs, tags, status.
Pull and persist inventory state at the InventoryItem × Location grain so multi-location stock can be reported.
Pull and persist subscription contracts where the store uses Shopify's native subscription primitives, plus a documented adapter pattern for third-party subscription apps (Recharge, Skio, Bold, Awtomic).
Surface session, visit, and conversion analytics through ShopifyQL where available, with a clearly documented fallback path to Google Analytics 4 / Shopify Admin reports.
Expose all of the above through three interfaces: a Flask-based internal dashboard, a versioned REST + GraphQL API, and an MCP server callable from Claude desktop / Claude Code.
Run cleanly on a developer laptop today; deploy to a generic container target tomorrow without code changes.
2.2 Out of Scope (v1)
Write-back operations against Shopify (order edits, inventory adjustments, refunds). v1 is read-only for safety; mutations are a v2 feature gated behind explicit per-tool authorization.
Customer-facing storefront features. The Storefront API and Shopify's Storefront MCP are not used in this project; this is an internal Admin-side tool.
BI/visualization tooling. Charts and dashboards in the Flask UI cover operational needs, but dedicated BI work (Looker, Metabase, Power BI) is downstream and connects to the same Postgres database.
Replacement for Shopify's native admin UI. Operators continue to do day-to-day operations in Shopify; this app is for cross-store reporting and AI access.

4.2 Component Responsibilities
Ingest layer: Receives webhook deliveries, performs HMAC verification, enqueues work to a background worker. Also exposes a synchronous on-demand fetch service for queries against data the cache does not cover. Writes are mediated through repositories — the ingest code does not call the ORM directly.
Persistence: Postgres database with a row-level `store_id` tenancy column on every table. Indexes on `(store_id, processed_at)`, `(store_id, sku)`, `(store_id, location_id)` for the most common access patterns.
Data layer (repository pattern): All database access funnels through repository classes (one per aggregate: Order, Product, Variant, Inventory, Customer, Subscription, Location, Analytics). Repositories are the only layer that imports SQLAlchemy. They expose intent-revealing methods like `find_orders_in_window(store_id, since, until)` rather than letting callers write queries.
Service layer: Use-case classes (e.g., `OrderQueryService`, `InventoryReportingService`, `SyncOrchestrator`) compose one or more repositories with the Shopify client. This is the only layer the dashboard, REST API, GraphQL gateway, and MCP server are allowed to call.
Serving — Dashboard: Flask Blueprints rendering server-side HTML (HTMX recommended for simple, low-build UI; React optional). Calls into the service layer; never imports ORM models directly.
Serving — API: Versioned REST endpoints under /api/v1/, plus a thin GraphQL gateway (Strawberry or Ariadne). Routes are thin: parse input, call a service, serialize output. Authentication via API tokens issued from a small admin UI.
Serving — MCP: Separate process running FastMCP. Re-uses the exact same service layer as the REST API — there is no MCP-specific SQL. Supports both stdio (for local Claude desktop) and HTTP (for remote MCP clients) transports.
Architectural rule
Only repositories may touch SQLAlchemy. Only services may touch repositories. Routes, MCP tools, and templates only touch services. This is enforced by import lint (a small pre-commit check that grep-fails any forbidden import path) and is the core invariant of the codebase.
5. Authentication and Authorization
5.1 Custom App per Store
Each of the three stores will host a Custom App. The app's Admin API access token is the credential used by this application's ingest and on-demand fetch layers.
Important platform change — January 2026
As of January 1, 2026, Shopify no longer allows new custom apps to be created from the Shopify Admin UI. New custom apps must be created via the Shopify Dev Dashboard or the Shopify CLI. Existing custom apps continue to function. If an app already exists in any of the three stores, keep using it; otherwise create the new app via the Dev Dashboard route.

5.3 Secret Storage
Per-store access tokens are stored encrypted at rest. In local development, secrets live in a .env file (gitignored). In any deployed environment, secrets are pulled from the host's secret manager (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault, or Render env vars) via a thin abstraction layer. The application must never log raw access tokens; logging middleware redacts any header named X-Shopify-Access-Token.
5.4 Webhook HMAC Verification
All incoming webhook requests are verified by computing HMAC-SHA256(body, app_shared_secret) and comparing to the X-Shopify-Hmac-Sha256 header using a constant-time compare (hmac.compare_digest). Verification happens before any other handler logic. Requests that fail verification return HTTP 401 and are not enqueued.
5.5 Authorization for the Internal API and MCP
Internal API: bearer-token authentication; tokens issued through a small Flask admin view, scoped to read-only by default and per-store optionally.
MCP server (stdio): no network auth needed — the local OS user controls launch.
MCP server (HTTP transport): bearer token in Authorization header, same token store as the REST API.
Audit log: every API and MCP tool invocation is logged with caller identity, store, tool/route, and parameters (sanitized).
6. Shopify API Surface
6.1 Admin API — GraphQL First
All new development uses the GraphQL Admin API. Per Shopify's October 2024 announcement, the REST Admin API is legacy, and as of April 2025 all new public apps must be GraphQL. Shopify continues to ship quarterly versions; as of this design's writing, 2026-04 is the latest stable release. The application pins to a specific quarter (e.g., 2026-04) and upgrades on a quarterly cadence.
6.2 Rate Limits
The GraphQL Admin API uses a cost-based leaky bucket. Each query reports `extensions.cost.actualQueryCost` and a `throttleStatus` block. The implementation must:
Inspect throttleStatus on every response and back off when `currentlyAvailable` drops below a threshold (e.g., 100 points).
Use exponential backoff with jitter on THROTTLED errors.
Track per-store cost consumption in a Prometheus/StatsD counter so the team can see saturation early.

6.3 Bulk Operations API
For nightly full reconciliation and any historical backfill, the Bulk Operations API is used instead of paginated queries. Bulk queries run asynchronously, are not subject to per-query cost limits, and produce a JSONL file at a temporary URL. Constraints:
Only one bulk operation per shop can run at a time — the scheduler must serialize.
Output must be downloaded promptly; URLs expire.
Bulk queries cannot use variables and have additional restrictions on connections and fragments.
6.4 ShopifyQL for Analytics
ShopifyQL is exposed via the GraphQL Admin API field `shopifyqlQuery`. It is the path to most session, visit, and conversion metrics that are otherwise only visible in the Shopify Admin Analytics UI. Available metrics include sessions, gross_sales, net_sales, orders, units_sold, customers, and revenue. Some metrics (average_order_value, conversion_rate, cart_abandonment_rate) cannot be ordered by time in cumulative calculations — these must be derived in application code.
Coverage caveat
ShopifyQL surface area has shifted multiple times since its preview. Some advanced features remain Plus-gated. Treat each metric the application depends on as a verification item: write a small probe query against each store and assert the result before relying on it in production code.
6.5 Python Client Choice
Recommendation: do not depend on shopify_python_api as the primary client. The package is still maintained but its GraphQL ergonomics lag the API and most of its surface area is REST-oriented. Instead, call the GraphQL endpoint directly with an HTTP client + GraphQL client:
httpx for HTTP — supports HTTP/2, retries, async; the Flask app uses httpx in sync mode and the worker can use it async.
gql (graphql-python/gql) for type-aware query construction and response validation.
ariadne-codegen or sgqlc optional, for auto-generating Python types from Shopify's introspected schema.
Endpoint pattern:
POST https://{shop}.myshopify.com/admin/api/2026-04/graphql.json
X-Shopify-Access-Token: {per-store token}
Content-Type: application/json

Webhook handler flow:

Read raw body (do not parse before HMAC verification).

Persist a `webhook_events_log` row with the raw payload (compressed).

Worker dequeues, normalizes the payload, and upserts into the relevant tables.
8.2 On-demand GraphQL fetch (synchronous fallback)
When the API or MCP receives a request for data the cache does not cover, or when a freshness flag in the request demands live data, the on-demand fetcher queries Shopify GraphQL directly. Results are written through to the cache to warm subsequent reads.
8.3 Nightly bulk reconciliation
Once per day, per store, the application launches a Bulk Operation that exports all orders modified in the last 48 hours plus a full snapshot of inventory levels. The output is reconciled against the cache to repair any rows that were missed due to webhook delivery failures (Shopify retries webhooks but eventually gives up after ~48 hours). This is the ground-truth reconciliation step that makes webhook-only sync trustable.
Why both webhooks and on-demand?
Webhooks alone are unreliable enough that production users do not trust them as the sole source. Bulk + nightly reconciliation closes the gap. On-demand is for ad-hoc queries that fall outside the synced surface (e.g., 'show me every order across all three stores from a specific date 18 months ago, including raw note attributes').
9. Analytics: Sessions, Visits, Conversion
Sessions, visits, and conversion are the most operationally important metrics that are not first-class objects in the Admin API. The application implements them in tiers.
9.1 Tier 1 — ShopifyQL via Admin GraphQL
Where ShopifyQL exposes the metric, query it nightly and write a row per store × day to `sessions_daily`. Sample query (executed via the `shopifyqlQuery` field):
{
  shopifyqlQuery(query: "FROM sales SHOW sessions, total_sales, orders SINCE -1d UNTIL -1d") {
    ... on TableResponse {
      tableData {
        rowData
        columns { name dataType }
      }
    }
    ... on ParseError { code message }
  }
}
9.2 Tier 2 — Google Analytics 4 fallback
If ShopifyQL coverage is insufficient or a metric is Plus-gated and the store is not on Plus, the application pulls sessions and conversion from the GA4 Data API for the property attached to that store. This requires a service-account credential per store; the discovery phase identifies whether GA4 is wired into each storefront.
9.3 Tier 3 — Computed metrics
Conversion rate, average order value, and other ratios are computed in the application from the lower-level metrics rather than queried, both because Shopify itself blocks some of them from time-ordered queries and because the team needs identical formulas across all three stores.
Discovery item
Before committing to ShopifyQL as the analytics path, write a probe query against each store and confirm `sessions` returns non-empty data over a known recent window. If any store fails, that store falls back to GA4.
10. Subscriptions
Shopify supports subscriptions via two distinct pathways:
Native: Apps that write to Shopify's `SubscriptionContract` GraphQL type. The Shopify Subscriptions app, Awtomic, and a growing number of others use this pathway. These appear directly in the Admin GraphQL API.
Third-party (off-platform state): Apps such as Recharge historically store contracts in their own database and expose their own API. These do not appear in `SubscriptionContract` queries.
10.1 Adapter Pattern
The application defines a single `SubscriptionProvider` interface with two implementations: `NativeProvider` (queries `SubscriptionContract` via Admin GraphQL) and one or more `ThirdPartyProvider` implementations (e.g., `RechargeProvider`). Each store is configured with the provider that matches its installed app. Both implementations write to the same `subscription_contracts` table so downstream consumers (dashboard, API, MCP) are app-agnostic.
Discovery item
Identify which subscription app is installed in each of the three stores and which pathway it uses. This determines which provider implementation must be built first and whether additional credentials (e.g., a Recharge API token per store) are required.

Repository interfaces are Python `Protocol` classes in `app/domain/repositories.py`. Concrete implementations live in `app/repositories/` and depend on SQLAlchemy. The domain layer never imports SQLAlchemy.

11A.3 Implementation in the data layer
# app/repositories/orders.py
from sqlalchemy.orm import Session
from app.domain.models import Order
from app.db.orm.order import OrderRow

class SqlAlchemyOrderRepository:
    """Implements domain.repositories.OrderRepository."""
    def __init__(self, session: Session):
        self._session = session

    def get(self, store_id, order_id):
        row = self._session.get(OrderRow, (store_id, order_id))
        return _row_to_domain(row) if row else None

    def find_in_window(self, store_id, since, until, financial_status=None,
                       sku=None, limit=50, cursor=None):
        q = (self._session.query(OrderRow)
                          .filter(OrderRow.store_id == store_id,
                                  OrderRow.processed_at >= since,
                                  OrderRow.processed_at < until))
        if financial_status:
            q = q.filter(OrderRow.financial_status == financial_status)
        if sku:
            q = q.join(OrderRow.line_items).filter(LineItemRow.sku == sku)
        if cursor:
            q = q.filter(OrderRow.processed_at < _decode_cursor(cursor))
        rows = q.order_by(OrderRow.processed_at.desc()).limit(limit + 1).all()
        next_cursor = _encode_cursor(rows[-1].processed_at) if len(rows) > limit else None
        return [_row_to_domain(r) for r in rows[:limit]], next_cursor

11A.4 Unit of Work
Transactions are managed by a `UnitOfWork` context manager that wraps a SQLAlchemy session and exposes the repositories. Services call repositories only inside a UoW block. Commit and rollback are explicit.
# app/services/order_query.py
class OrderQueryService:
    def __init__(self, uow_factory):
        self._uow_factory = uow_factory

    def list_orders(self, store_id, since, until, **filters):
        with self._uow_factory() as uow:
            orders, next_cursor = uow.orders.find_in_window(
                store_id, since, until, **filters
            )
            return ListOrdersResult(items=orders, next_cursor=next_cursor)

11A.5 Domain objects, not ORM rows
Repositories return immutable domain objects (Python dataclasses with `frozen=True` or pydantic models). ORM rows never escape the data layer. Reasons:
Lazy-loaded SQLAlchemy attributes can issue surprise queries hours later, in unrelated code paths. Returning a fully-materialized dataclass eliminates this class of bug.
Domain objects can be constructed from non-DB sources (live Shopify GraphQL responses, in-memory test fixtures) without changing service code.
Serialization for the API and MCP layers is uniform — both sides see the same domain types.
11A.6 Specifications for complex queries
Where filtering grows beyond a handful of optional parameters (Order search is the obvious example), use a `Specification` object instead of more keyword args:

Both the Flask app factory and the MCP entrypoint pull pre-built service instances from the container. Tests build their own container with `InMemoryOrderRepository` (etc.) for fast service-layer unit tests.

11A.9 Enforcement
The architectural invariants are enforced by tooling rather than code review:
A pre-commit hook runs `ruff check --select TID` with import-tidy rules forbidding `app.db` imports outside `app.repositories.*`, forbidding `sqlalchemy` imports outside `app.repositories.*` and `app.db.*`, and forbidding `app.repositories.*` imports outside `app.services.*` and `app.container`.
`tests/architecture/` contains import-graph tests that fail CI if the layer rules are violated.
Services accept repository protocols, not concrete classes — the static type-check (mypy / pyright) catches accidental coupling.

13. MCP Server Design
The MCP server lets Claude (and other compatible LLM clients) ask natural-language questions across the three stores. It is a separate process from Flask so it can run independently — for example, alongside Claude Desktop on a developer machine via stdio, or behind a reverse proxy in HTTP mode for shared use.
13.1 Why a custom MCP and not Shopify's
Shopify currently ships four official MCP servers — Dev, Storefront, Customer Account, and Checkout — plus the Universal Commerce Protocol catalog. None of these expose internal Admin-side reporting across multiple stores: the Storefront MCP is for buyer-facing assistants, Customer Account is per-customer with OAuth, Dev is for assisting developers with Shopify schema queries, and Checkout is for the checkout flow. The internal use case here — "summarize last week's orders across all three stores" — is not in scope for any of them. The application therefore ships its own MCP that wraps its own Postgres cache and on-demand fetch path.
13.2 Framework
FastMCP (Python). Comparable ergonomics to Flask, supports both stdio and HTTP transports out of the box, integrates with pydantic for typed tool schemas.

13.4 Tool design conventions
Each MCP tool is a thin function that constructs a service call and serializes the result. The service classes are the same ones the REST API uses; there is no MCP-specific data access layer. New capabilities are added once at the service layer and surfaced in both the API and MCP simultaneously.
Every tool input/output is a pydantic model with field descriptions — these become the LLM-visible schema.
Date inputs accept ISO 8601 strings or relative phrases like 'yesterday', '7d', 'last_week'. The MCP layer normalizes before passing to the data layer.
Result sets are paginated; tools return at most ~50 rows per call and surface a `next_cursor` for follow-up calls.
Responses include a `freshness` field per row indicating cache age in seconds, so the LLM can choose to call refresh_order when needed.
Mutations are out of scope for v1. v2 may add `pause_subscription`, `adjust_inventory`, etc., each behind a per-tool authorization scope.

15. Portability and Deployment
The application is built to 12-factor conventions so it can move from laptop to any container target without code changes.
Configuration: every external dependency (database URL, Redis URL, per-store credentials, webhook tunnel URL, GA4 credentials) is read from environment variables.
Stateless web layer: Flask processes can be killed and replaced without losing state. All state is in Postgres or Redis.
Idempotent migrations: Alembic only.
Containerized: a single Dockerfile builds an image with both the Flask app and the worker entrypoint; orchestration (docker-compose, Kubernetes, Render Blueprint) selects which command to run.
Secret abstraction: a thin `secrets` module reads from env in dev and from a configurable backend (AWS Secrets Manager, GCP, Vault) in production.
Logging: structured JSON to stdout; aggregation is the host's job.

16. Security and Compliance Considerations
Webhook HMAC verification on every inbound delivery, with constant-time compare and request-body capture before parsing.
Per-store access tokens encrypted at rest, never logged, never returned by any API endpoint.
PII (customer names, addresses, emails, phones) is restricted to specific endpoints; the dashboard masks email/phone by default and reveals on click with audit trail.
API tokens for internal callers expire on a configurable cadence (default 90 days).
MCP HTTP transport is bearer-token gated and intended for trusted networks; not exposed publicly without an authentication proxy.
Rate limiting on the internal API and MCP HTTP endpoint to prevent runaway clients from saturating the Shopify cost budget.
Dependencies pinned and scanned (pip-audit / Dependabot).
Backups: managed Postgres provider performs point-in-time recovery; nightly export of normalized data to object storage as a defense-in-depth measure.
17. Observability
Structured JSON logs with: request_id, store_id, route, latency_ms, status, query_cost (for Shopify calls).
Per-store dashboards: webhook delivery success rate, queue depth, GraphQL cost consumption, sync lag (max(now() - last_seen) per resource).
Alerts: webhook auth failures > 0 in 5 min, queue depth > N for 10 min, sync lag > 30 min on orders, GraphQL throttle errors > 5% of calls.
Tracing optional in v1; instrument with OpenTelemetry hooks so it can be turned on without code changes.
MCP-specific: log tool name, caller, latency, and a hash of inputs (not full inputs) so the team can analyze usage patterns without storing query content.
